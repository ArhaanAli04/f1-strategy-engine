# /run-day — Start a day's build session

Use this command at the beginning of every daily work session.

## What to do

1. Read CLAUDE.md in full if you have not already done so this session.

2. Run these orientation commands before touching any code:
   - `git log --oneline -10` to see recent commits
   - `git status` to see any uncommitted changes
   - `find backend/ -name "*.py" | sort` to see current file inventory
   - `cat CLAUDE.md | grep -A 5 "Current Project Phase"` to read today's context

3. Ask the user: "Confirm today is Day [X] and the goal is [Y]. 
   Should I proceed with implementation or do you want to adjust anything first?"

4. Wait for confirmation before writing any file.

5. After confirmation: implement each bullet point from the day's timeline 
   one at a time. After each file is written, run the relevant check:
   - Python file → `cd backend && python -c "import <module>"` (syntax check)
   - Migration → run it against a live local database if available and 
     verify it actually applied (do not assume a hand-written or 
     autogenerated migration is correct without running it)
   - Service method → run the corresponding unit test if it exists
   - API endpoint → confirm it appears in `GET /docs`

6. After all files for the day are written:
   - Run `make lint` — fix all errors before proceeding
   - Run `make test-unit` — all must pass (0 collected is expected before 
     Day 14, that is not a failure)
   - Summarise what was built today in a clear list (file by file, one 
     line each) so the user can review before anything is committed

## STOP HERE — hand control back to the user

Do NOT run `git add`, `git commit`, or any other git command beyond the 
read-only ones in Step 2 (`git log`, `git status`). Do NOT edit CLAUDE.md, 
including the "Current Project Phase" block.

These steps are intentionally reserved for the user, every single day, 
without exception:
- Staging and committing today's work
- Pushing the branch
- Opening and merging the Pull Request
- Updating the "Current Project Phase" block in CLAUDE.md
- Committing and pushing that CLAUDE.md update

If the user explicitly asks you to run `/pre-commit` after this point, 
that command may run git add and git commit as part of its own defined 
steps — but even then, do not touch CLAUDE.md's phase tracker, and do 
not push, open PRs, or merge anything. Those remain the user's steps 
regardless of which command is running.

## Usage
Paste the day's bullet points from the timeline doc after invoking this command.
